# Skyinfii 个人网站主页设计说明

## 1. 设计结论

新的 `https://www.skyinfii.site/` 不应继续作为知识库首页，而应改为 **工程师的公开工作台**。

知识库仍然是最重要入口，但它不再独占首页身份。首页承担更高一层的信息架构职责：让访问者快速知道你是谁、你在构建什么、可以从哪里进入知识库、作品、实验和联系方式。

当前 `/wiki/` 可以保留为知识库子站；新的 `/` 作为个人导航主页。

## 2. 当前问题

现站首页抓取结果显示，当前页面主要由以下模块组成：

- 顶部品牌：`知识库`
- 导航：搜索、技术概念、项目分析、工具手册、架构师备考、学习方法论
- Hero：个人知识库说明与搜索入口
- 统计：笔记总数、主题数、Markdown、HTML、未解析链接、构建来源、最近同步
- 学习路径、栏目浏览、热门标签、最近更新

这些内容对知识库很有价值，但作为个人主页存在三个问题：

1. **身份单一**：访客只能感知“这是一个知识库”，无法理解你的个人能力、项目和正在构建的系统。
2. **入口层级过低**：首页直接进入知识库内部结构，缺少更高层级的导航。
3. **作品承载不足**：SkillHub、前端 AI 设计框架、知识库展示平台等项目没有获得独立入口。

## 3. 主页定位

### 核心定位

**工程师的公开工作台**

### 一句话说明

我在这里整理前端工程、架构学习、AI 工具链和个人知识系统。这里不是一份静态简历，而是一张可以进入的工作地图。

### 目标用户

| 用户 | 来访意图 | 首页应提供 |
|---|---|---|
| 同行 / 朋友 | 快速了解你在做什么 | 清晰定位、代表项目、知识库入口 |
| 潜在合作方 | 判断能力和方向是否匹配 | 项目入口、作品说明、联系方式 |
| 未来的自己 | 快速进入常用内容 | 知识库、实验、最近更新 |
| 搜索访问者 | 从文章进入后探索更多 | 主页回流入口、相关项目入口 |

## 4. 信息架构

### 一级导航

```text
Skyinfii
├── 知识库
├── 作品
├── 实验
└── 关于
```

### 入口职责

| 入口 | 目标路径 | 职责 |
|---|---|---|
| 知识库 | `/wiki/` | 进入现有 Obsidian / Astro 知识库 |
| 作品 | `/projects/` 或首页锚点 | 展示已经形成成果的项目 |
| 实验 | `/lab/` 或首页锚点 | 展示正在试跑、未完全产品化的探索 |
| 关于 | `/about/` 或首页底部 | 简短介绍、联系方式、社交链接 |

MVP 阶段可以先不创建独立 `/projects/`、`/lab/`、`/about/` 页面，而是在首页内用锚点实现。

## 5. 首页结构

### 5.1 Header

职责：提供稳定导航，不抢首屏视觉。

内容：

- 左侧：`Skyinfii`
- 右侧：`知识库`、`作品`、`实验`、`关于`
- 可选：主题切换保留，但不作为首屏重点

设计要求：

- 桌面端横向导航
- 移动端折叠为简单菜单或横向滚动
- 不复用知识库内部过多栏目导航，避免首页变成二级栏目页

### 5.2 Hero

职责：在 5 秒内说明“这里是什么”。

推荐文案：

```text
公开的工程工作台

我在这里整理前端工程、架构学习、AI 工具链和个人知识系统。
这里不是一份静态简历，而是一张可以进入的工作地图。
```

主行动：

1. `进入知识库`
2. `查看作品`
3. `联系我`

设计要求：

- 首屏应明确出现 `Skyinfii` 或个人站身份
- Hero 不使用“个人知识库”作为唯一标题
- 不用大段自我介绍，先给入口

### 5.3 Now Building

职责：让主页有“正在发生”的生命感。

建议展示 3 个当前重点：

| 条目 | 说明 |
|---|---|
| 前端 AI 设计框架 | 把设计前置为可执行架构约束 |
| SkillHub | 管理和试跑可复用 skill 的工具 |
| 知识库公开化 | Obsidian -> Astro -> 可搜索静态站 |

展示方式：

- 右侧状态面板
- 每项一行：项目名 + 一句话说明
- 可以加状态：`active`、`draft`、`shipping`

### 5.4 Portal Cards

职责：把首页变成真正的导航枢纽。

三张主入口：

| 卡片 | 说明 | 行动 |
|---|---|---|
| 知识库 | 技术概念、项目分析、工具手册、学习路径 | 进入 `/wiki/` |
| 作品 / 项目 | SkillHub、前端 AI 设计框架、知识库展示平台 | 查看项目 |
| 实验室 | AI workflow、架构试跑、设计系统、自动化脚本 | 浏览实验 |

设计要求：

- 知识库卡片权重最高
- 作品和实验不能只是小链接，应作为一级入口
- 卡片文案要具体，避免“更多内容”“探索一下”这类空泛表达

### 5.5 Featured Projects

职责：证明“工作台”不是口号。

MVP 推荐展示 3 个项目：

| 项目 | 类型 | 推荐描述 |
|---|---|---|
| 前端 AI 设计框架 | 方法论 / Skill | 用结构化流程约束 AI 前端设计，避免直接 vibe coding |
| SkillHub | 工具 / 平台 | 管理、试跑、验收可复用 skill 的工作台 |
| 知识库展示平台 | 内容系统 | 将 Obsidian 笔记发布为可搜索、可浏览的静态知识站 |

每个项目包含：

- 项目名
- 一句话价值
- 状态：`active` / `prototype` / `published`
- 链接：文档、demo、仓库或知识库文章

### 5.6 Knowledge Paths

职责：保留现有知识库优势，但不要让路径列表压过主页主线。

建议保留 4 条路径：

- 前端工程化路径
- AI 工具链路径
- 个人知识管理路径
- 架构师备考路径

展示规则：

- 首页只放精选路径
- 更多路径回到 `/wiki/`
- 不在首页展示全部标签云，标签云留在知识库内部

### 5.7 Recent Updates

职责：证明站点持续更新。

MVP 展示 4-6 条即可：

- 标题
- 类型：笔记 / 项目 / 实验
- 更新时间
- 简短摘要

不建议在主页展示过多标签，避免视觉噪音。

### 5.8 Contact / About

职责：提供最低限度的联系和个人说明。

内容：

- 一段短介绍
- GitHub / Email / 其他社交入口
- 可选：`如果你对前端工程、AI workflow、知识管理感兴趣，可以联系我。`

## 6. 视觉方向

### 关键词

```text
技术感、可进入、持续构建、克制但有识别度
```

### 色彩策略

采用深色工作台基底 + 高亮行动色：

| 角色 | 建议 |
|---|---|
| 页面背景 | 接近黑绿 / 深墨色，承载“工作台”氛围 |
| 主文字 | 近白，保证可读性 |
| 辅助文字 | 降低透明度，但必须满足可读性 |
| 强入口色 | 青绿色，用于知识库主按钮和状态点 |
| 次强调色 | 酸黄绿色，用于作品 / 实验卡片的差异化 |

不要使用：

- 默认浅灰卡片网格
- 大面积紫蓝渐变
- 纯知识库文档站式白底列表
- 过多阴影和玻璃拟态

### 字体与排版

原则：

- 中文正文优先系统字体，保证加载速度和稳定性
- Hero 标题可以使用更强的字重和紧凑行高
- 正文行长控制在 65-75 字符
- 标题使用 `text-wrap: balance`
- 不使用到处都是的 tiny uppercase eyebrow

Hero 标题建议：

```css
font-size: clamp(2.75rem, 8vw, 5.25rem);
line-height: 0.96;
letter-spacing: -0.03em;
```

### 布局节奏

首屏建议使用两栏：

```text
左：定位文案 + 主行动
右：Now Building 状态面板
下：三张入口卡片
```

移动端改为单列：

```text
定位文案
主行动
Now Building
入口卡片
精选项目
最近更新
```

## 8. 技术与架构建议

### 推荐技术栈

| 维度 | 选择 | 理由 |
|---|---|---|
| 框架 | Astro | 当前知识库站点已经是 Astro 静态站，适合 Content / 个人主页 |
| 组件 | Astro components + 原生 HTML | Level 1 内容站不需要重型组件库 |
| 样式 | CSS Variables + scoped CSS | 足够可控，避免引入多余 runtime |
| 搜索 | 沿用 Pagefind 于 `/wiki/` | 首页只提供搜索入口，不重复搜索系统 |
| 状态 | 不使用全局状态 | 首页静态导航，无复杂状态 |
| 图标 | 内联 SVG 或少量本地图标 | 避免为几个图标引入完整图标库 |

### 目录建议

在现有 Astro 项目中建议形成以下结构：

```text
src/
├── pages/
│   ├── index.astro              # 新个人主页
│   └── wiki/                    # 保留现有知识库路由

│   └── WikiLayout.astro         # 知识库布局
├── components/

│   │   ├── NowBuildingPanel.astro
│   │   ├── PortalCardGrid.astro

│       ├── SiteHeader.astro

```

## 9. MVP 范围

### P0 必须完成

| 项 | 内容 |
|---|---|
| 新首页 `/` | 不再直接等同于 `/wiki/` |
| Hero | 标题、说明、三个主行动 |
| Now Building | 3 个当前构建项 |
| Portal Cards | 知识库、作品、实验三入口 |
| Featured Projects | 至少 3 个代表项目 |
| 基础响应式 | 移动端单列可读 |

### P1 可补充

| 项 | 内容 |
|---|---|
| Knowledge Paths | 精选 4 条学习路径 |
| Recent Updates | 从知识库最近更新中选 4-6 条 |
| ContactBlock | 联系方式与简短介绍 |
| 主题切换 | 沿用现有 wiki 主题能力或只保留深色首页 |

### P2 可延后

| 项 | 延后理由 |
|---|---|
| 独立 `/projects/` 页面 | MVP 可先用首页区块承载 |
| 独立 `/lab/` 页面 | 实验内容未必已足够多 |
| 自动聚合项目数据 | 初期手写更快、更可控 |
| 首页搜索 | 搜索已在 `/wiki/` 中存在 |

### 不可接受的技术债

- `/` 和 `/wiki/` 语义继续混在一起
- 首页只换皮但信息架构仍然是知识库栏目页
- 移动端导航不可用
- 主行动链接不明确或不可点击
- 项目入口没有真实目标 URL 或占位说明

## 10. 文案建议

### Hero 标题备选

推荐：

```text
公开的工程工作台
```

备选：

```text
我的工程与知识地图
```

```text
把学习、工具和项目放到一张桌面上
```

不建议：

```text
个人知识库
```

原因：它会把主页重新拉回现有问题。

### Portal 卡片文案

```text
知识库
技术概念、项目分析、工具手册、学习路径。进入可搜索的公开笔记系统。

作品 / 项目
SkillHub、前端 AI 设计框架、知识库展示平台，以及持续构建中的工具。

实验室
AI workflow、架构试跑、设计系统和自动化脚本。这里放还在生长的东西。
```

## 11. 后续编码约束

1. 先创建新的 `/` 首页，不破坏现有 `/wiki/`。
2. 首页组件放在 `components/home/`，不要混入 wiki 内页组件。
3. 不引入 Ant Design、MUI、shadcn/ui 等重型或不必要组件库。
4. 首页样式使用 CSS Variables 和 scoped CSS，避免散落硬编码。
5. P0 阶段首页内容可手写，但数据集中在 `src/data/home.ts`。
6. 所有导航入口必须有真实目标或明确占位策略。
7. 首页只展示精选内容，知识库的完整栏目、标签云、搜索页继续留在 `/wiki/`。

## 13. 推荐下一步

1. 在真实站点仓库中新增或修改 `src/pages/index.astro`。
2. 创建 `HomeLayout.astro` 与 `components/home/*`。
3. 将首页内容数据集中到 `src/data/home.ts`。
4. 保留 `/wiki/` 作为知识库子站。
5. 完成后用浏览器检查桌面端和移动端首屏，重点看导航是否清晰、入口是否可点击、文字是否溢出。
