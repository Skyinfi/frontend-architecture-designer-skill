# 模块边界与领域拆分参考

> 本文件供 `core/workflow.md` §6（模块边界与领域拆分）查表使用。
> 仅在 **Level 2+** 输出时参考。Level 1 项目不需要领域拆分。

---

## 一、什么时候需要领域拆分

| 信号 | 需要拆分 | 不需要拆分 |
|---|---|---|
| 页面数 | > 30，涉及多条业务线 | < 15，业务线单一 |
| 团队 | 多人/多组，各自负责不同功能 | 1–2 人，全栈维护 |
| 功能独立性 | 各功能模块几乎独立演进 | 功能高度耦合，改一处动全局 |
| 数据模型 | 不同业务有各自的数据实体和 API | 共享少量数据模型 |
| 复杂度 Level | Level 2（简）或 Level 3（完整） | Level 1（不需要） |

**核心原则**：不强行给小项目套 DDD。领域拆分是为了解决**复杂度管理**问题，不是为了"架构正确"。

---

## 二、领域类型判断

### 核心域（Core Domain）

**定义**：项目存在的理由，直接创造业务价值的功能。

**识别方法**：
- 去掉这个模块，项目是否失去了核心价值？
- 用户付费/使用的主要原因是否来自这个模块？
- 这个模块的业务逻辑是否最复杂、最不可替代？

**前端示例**：
| 项目类型 | 核心域 |
|---|---|
| 电商后台 | 商品管理、订单管理、营销活动 |
| SaaS Dashboard | 数据看板、报告生成、告警配置 |
| 内容平台 | 内容编辑、发布流程、内容审核 |
| 知识库工具 | 笔记管理、知识图谱、搜索 |

### 支撑域（Support Domain）

**定义**：不直接创造价值，但支撑核心域运行的功能。

**识别方法**：
- 核心域能否独立运行？如果不行，需要什么支撑？
- 这些功能是否在多个项目中反复出现？

**前端示例**：
| 支撑域 | 说明 |
|---|---|
| 通知系统 | 站内信、消息推送 |
| 文件上传 | 图片/附件/文档上传 |
| 搜索 | 全文搜索、过滤、排序 |
| 数据导出 | CSV/Excel/PDF 导出 |
| 工作流 | 审批流程、状态机 |

### 通用域（Shared Domain）

**定义**：与业务无关的基础能力，几乎所有项目都需要。

**前端示例**：
| 通用域 | 说明 |
|---|---|
| 认证授权 | 登录、注册、权限、角色 |
| 主题系统 | Light/Dark/System 切换 |
| 国际化 | i18n、多语言、日期格式 |
| 请求层 | HTTP 封装、错误处理、拦截器 |
| 通用 UI | Button、Modal、Toast、Form 组件 |

## 三、模块间依赖规则

```
核心域 → 支撑域 → 通用域
（依赖方向：从左到右，不可反向）
```

| 规则 | 说明 |
|---|---|
| 核心域可依赖支撑域和通用域 | 核心业务可以使用支撑和基础能力 |
| 支撑域可依赖通用域 | 支撑功能可以使用基础能力 |
| 通用域不依赖核心域或支撑域 | 基础能力不应感知业务 |
| 核心域之间尽量不互相依赖 | 不同的核心业务应独立演进 |

**循环依赖处理**：
- 如果 A 依赖 B 且 B 依赖 A → 提取共享部分到通用域
- 如果是事件驱动的松耦合 → 通过事件总线 / 状态管理解耦，不直接 import

## 四、Register 差异化建议

### Content Register

**通常不需要领域拆分**。内容站的核心是内容本身，模块边界主要体现在：

| 模块 | 边界 | 说明 |
|---|---|---|
| 内容渲染 | 独立 | Markdown / HTML 渲染管线 |
| 内容导航 | 独立 | 目录、分类、标签 |
| 搜索 | 可选 | Pagefind 等静态搜索 |
| 构建管线 | 独立 | sync 脚本、预处理 |

这些模块通过 Astro 约定目录自然隔离，不需要显式的领域建模。

### Tool Register

**Level 2 需要简单的 feature-based 拆分，Level 3 需要领域化拆分。**

常见功能模块划分：

| 模块 | 域类型 | 典型内容 |
|---|---|---|
| 用户管理 | 通用域 | 登录、注册、权限、角色 |
| 数据管理 | 核心域 | CRUD 表格、表单、详情页 |
| 工作流 | 支撑域 | 审批、状态流转 |
| 报表/统计 | 支撑域 | 数据看板、图表 |
| 系统设置 | 通用域 | 配置管理、字典管理 |

### Desktop Register

**按功能面板拆分**，通常不需要领域化：

| 模块 | 边界 |
|---|---|
| 主面板 | 核心交互区域 |
| 侧边栏 | 导航和工具 |
| 设置面板 | 配置和偏好 |
| 通知区域 | 系统通知 |
| 本地集成 | 文件系统、系统托盘 |

### Hybrid Register

**按页面类型拆分**，不同类型页面遵循不同的模块组织方式：

- 内容页 → Content Register 的轻量模式
- 工具页 → Tool Register 的 feature-based 模式
- 共享部分 → 提取到通用模块

## 五、前端领域边界实操方法

### Step 1：识别实体

列出系统中的主要数据实体（名词）：

```
商品、订单、用户、权限、文章、分类、标签...
```

### Step 2：识别动词

列出用户能做的主要操作（动词）：

```
创建商品、提交订单、审核文章、导出报表...
```

### Step 3：按实体聚合

将动词归到对应实体下：

```
商品：创建、编辑、上下架、删除、批量导入
订单：创建、支付、发货、取消、退款
```

### Step 4：确定域类型

- 付费/使用的核心流程 → 核心域
- 辅助核心流程的 → 支撑域
- 所有项目都有的 → 通用域

### Step 5：定义模块 API

每个模块通过 `index.ts` 暴露公共接口：

```typescript
// features/product/index.ts
export { ProductList } from './components/ProductList'
export { ProductForm } from './components/ProductForm'
export { useProduct } from './hooks/useProduct'
export type { Product, ProductFilter } from './types'
```

其他模块只通过 `index.ts` 消费，不直接 import 内部文件。

## 六、常见反模式

| 反模式 | 为什么是错的 | 正确做法 |
|---|---|---|
| 对 Level 1 项目做领域拆分 | 增加认知负担，没有收益 | 小项目用扁平结构 |
| 对简单 CRUD 做完整 DDD | CRUD 不需要领域建模 | feature-based 足够 |
| 所有模块平铺在 `src/` 根下 | 找不到东西，依赖混乱 | 按功能/领域分目录 |
| 核心域之间互相 import | 耦合严重，改一处动全局 | 通过事件/状态解耦，或提取共享到通用域 |
| 通用域包含业务逻辑 | 违反依赖方向原则 | 通用域只放纯技术能力 |
| `index.ts` 导出内部实现细节 | 破坏了模块封装 | 只导出公共 API（组件、hooks、类型） |
| 过度拆分：一个功能拆成 5 个模块 | 增加间接层，降低开发效率 | 按业务内聚性拆分，不是越细越好 |
| 不拆分：一个 100+ 文件的 `features/` | 失去了拆分的意义 | 当 feature 内部文件 > 15 时考虑进一步拆分 |

## 七、Level 与拆分策略

| Level | 拆分策略 | 输出内容 |
|---|---|---|
| **Level 1** | 不拆分 | 标注"本项目复杂度较低，此项暂不展开" |
| **Level 2** | Feature-based（简） | 给出功能模块列表 + 依赖方向，不展开内部结构 |
| **Level 3** | 领域化（完整） | 核心域/支撑域/通用域划分 + 模块 API + 依赖规则 + 循环依赖处理 |
