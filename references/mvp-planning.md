# MVP 规划参考

> 本文件供 `output-template.md` §11（MVP 第一阶段范围）查表使用。
> **所有 Level 都需要 MVP 规划**，但深度不同。

---

## 一、MVP 范围判断原则

### 核心原则

1. **MVP 是可验证的最小价值单元**，不是"功能砍半"
2. **MVP 必须包含完整的用户价值闭环**，不是半成品
3. **MVP 架构决策应覆盖所有 Level**，技术债可以有但要知道在哪
4. **MVP 不是偷懒的借口**，该做的架构基础不能省

### 判断流程

```
1. 列出所有期望功能
2. 标记"没有这个功能产品就不成立"的项 → 必须
3. 标记"有了更好但没有也能跑"的项 → 延后
4. 在"必须"项中找出最小组合 → MVP
5. 检查 MVP 是否构成完整闭环
```

## 二、页面范围判断

### 必须页面的判断标准

| 标准 | 说明 |
|---|---|
| 核心用户流程必须经过此页面 | 如：列表→详情，缺了就不成立 |
| 没有此页面无法验证核心价值 | 如：内容站必须能查看内容 |
| 此页面是后续所有功能的基础 | 如：布局框架、路由系统 |

### 延后页面的判断标准

| 标准 | 说明 |
|---|---|
| 独立功能，不影响核心流程 | 如：设置页、个人中心 |
| 需要额外数据/服务支持 | 如：需要后端 API 的统计页 |
| 属于体验优化而非核心价值 | 如：搜索建议、高级过滤 |

### Register 差异

| Register | MVP 必备页面 | 可延后 |
|---|---|---|
| Content | 内容详情页、栏目列表页、首页 | 搜索页、Tag 过滤、关联推荐、深色主题 |
| Tool | 登录页、主列表/表格页、表单页、详情页 | 统计看板、导出功能、高级筛选、批量操作 |
| Desktop | 主面板、核心交互区 | 设置页、命令面板、系统托盘、自动更新 |
| Hybrid | 按主要页面类型的必备页面 | 次要页面类型的全部 |

## 三、组件优先级

### 必须封装的组件

| 判断标准 | 说明 |
|---|---|
| 多个 MVP 页面都用到 | 如 AppShell、基础布局 |
| 不封装会导致页面代码混乱 | 如 DataTable、Form |
| 封装后能建立稳定模式 | 如 API 请求 hook、错误处理 |

### 可延后封装的组件

| 判断标准 | 说明 |
|---|---|
| 只在 1 个页面用到 | 内联即可 |
| 属于锦上添花的 UI | 如动画组件、高级图表 |
| 需要额外依赖 | 如富文本编辑器、拖拽排序 |

## 四、技术债管理

### 技术债分类

| 类型 | 定义 | MVP 态度 |
|---|---|---|
| **可接受的技术债** | 已知问题，MVP 阶段不影响核心功能 | 记录，延后清理 |
| **不可接受的技术债** | 影响核心功能或安全的问题 | 必须在 MVP 解决 |

### 可接受的技术债示例

| 技术债 | 接受理由 | 清理时机 |
|---|---|---|
| 无测试覆盖 | MVP 先验证产品可行性 | 产品验证后补测试 |
| 硬编码文案 | 后续做 i18n 时统一提取 | i18n 启动时 |
| 无错误边界 | MVP 阶段页面简单，错误可控 | 页面复杂度增加时 |
| 简单的 loading 状态 | 功能优先 | P1 加骨架屏 |
| 无 a11y 处理 | 产品验证优先 | P1 补基础 a11y |

### 不可接受的技术债示例

| 技术债 | 为什么不可接受 | 正确做法 |
|---|---|---|
| 没有错误处理 | 用户看到白屏 | 至少有全局错误兜底 |
| 绕过架构方案直接编码 | 架构方案白写了 | 按方案执行 |
| 不按目录结构组织文件 | 后续维护困难 | 按方案创建目录 |
| 组件不按分层规则写 | 混乱累积，越来越难改 | 遵循三层分离 |
| 安全漏洞（XSS、token 明文） | 上线即风险 | MVP 也要基本安全 |

### 技术债记录方式

在 `frontend-architecture.md` 的 MVP 章节中维护：

```markdown
### 可接受的技术债
| 技术债 | 接受理由 | 清理时机 |
|---|---|---|
| 无测试覆盖 | 先验证产品 | P1 |

### 不可接受的技术债
- 必须有全局错误处理
- 必须按目录结构组织
```

## 五、Level 与 MVP 深度

### Level 1：轻量 MVP

**页面范围**：通常 2–4 个核心页面

| 必须 | 说明 |
|---|---|
| 核心页面 | 产品存在理由的页面 |
| 基础布局 | AppShell / Layout |
| 基础组件 | 项目中重复出现 3+ 次的结构 |

**延后**：搜索、主题切换、高级交互、响应式完善

**输出格式**：
- 必须完成的页面表
- 必须封装的组件表
- 明确延后的能力表
- 可接受/不可接受的技术债表

### Level 2：标准 MVP

**页面范围**：核心流程的全部页面（5–10 个）

在 Level 1 基础上增加：
- 统一错误处理
- 基础 loading / empty / error 状态组件
- 请求层封装
- 基础测试覆盖（核心流程）

### Level 3：完整 MVP

**页面范围**：所有主要业务流程的完整闭环

在 Level 2 基础上增加：
- 完整 Design Token
- 主题切换
- 完整测试策略
- 性能 / bundle 预算
- a11y 基线

## 七、Register 差异化建议

### Content Register

| 应该 | 不应该 |
|---|---|
| MVP 只做内容详情 + 列表 | 一开始就做搜索、Tag 过滤 |
| 构建管线优先于页面设计 | 先做漂亮首页再做内容渲染 |
| 接受 iframe 双层滚动等技术债 | 为了完美体验推迟 MVP |

### Tool Register

| 应该 | 不应该 |
|---|---|
| MVP 包含完整的 CRUD 流程 | 只做列表不做表单 |
| 表格/表单用组件库现成方案 | 自己造轮子 |
| 权限框架在 MVP 就建立 | 等后期再补权限 |

### Desktop Register

| 应该 | 不应该 |
|---|---|
| MVP 先做核心面板 | 一开始就做命令面板、自动更新 |
| 本地功能优先于网络功能 | 先做云同步再做本地功能 |
| 关注窗口管理和快捷键 | 忽略桌面端特有的交互模式 |

### Hybrid Register

| 应该 | 不应该 |
|---|---|
| 按主要页面类型规划 MVP | 要求所有页面类型都做到 P0 |
| 内容页和工具页分别规划优先级 | 一刀切 |

## 八、与既有 MVP 流程对比

当存量项目已有 MVP/迭代规划时，参考本文件的价值是**校验和补全**，而非推倒重来。

### 校验清单

| 检查项 | 健康信号 | 问题信号 |
|---|---|---|
| MVP 范围 | 核心用户流程闭环 | 核心流程不完整（只做列表不做详情） |
| 页面优先级 | 明确 P0/P1/P2 分期 | 所有页面同等对待 |
| 组件封装 | 按 3+ 复用原则封装 | 为每个页面单独封装组件 |
| 技术债 | 已记录 + 清理时机 | 凭印象延期，技术债被遗忘 |
| 架构约束 | MVP 就遵守三层分离 + 目录结构 | MVP 阶段绕过架构直接编码 |
| 错误处理 | 至少全局错误兜底 | 完全没有错误处理 |

### 补充建议

- 如果核心流程不闭环，优先补全核心页面，再考虑优化
- 如果没有 P0/P1/P2 分期，建议对 MVP 范围做优先级排序
- 如果技术债没记录，建议建立 `frontend-architecture.md` 中的"技术债"小节
- 如果 MVP 阶段绕过架构（如页面包含业务逻辑），建议在下一迭代重构成业务组件
- 如果没有错误处理，建议至少加 `ErrorBoundary` 全局兜底

## 九、常见反模式

| 反模式 | 为什么是错的 | 正确做法 |
|---|---|---|
| MVP = 偷工减料 | MVP 是最小完整产品，不是残缺品 | 确保核心价值闭环完整 |
| MVP 不需要架构方案 | 越是快速迭代越需要架构约束 | 架构方案是 MVP 的一部分 |
| MVP 做完整设计系统 | 过度设计，延误交付 | Level 1 只做基础 Token，其余延后 |
| "先跑起来再说"无技术债记录 | 技术债被遗忘，累积成定时炸弹 | 每个技术债都记录理由和清理时机 |
| MVP 做所有页面 | 范围膨胀，交付周期拉长 | 严格区分 P0/P1/P2 |
| 为 MVP 重写已有可用方案 | 浪费时间 | 存量项目增量改造 |
| 把"后续优化"当借口逃避核心决策 | 核心决策不能延后 | 架构约束、数据流、分层规则在 MVP 就确定 |
| MVP 不考虑性能 | 性能问题后期修复成本极高 | 至少设 bundle 预算和首屏指标 |

## 十、MVP 自检清单

输出 MVP 范围后，用以下清单自检：

| 检查项 | 通过标准 |
|---|---|
| 核心用户流程是否闭环？ | 从入口到目标结果，每一步都有页面 |
| MVP 页面是否覆盖核心价值？ | 去掉任一页面，产品是否仍有价值 |
| 技术债是否全部记录？ | 每个已知妥协都有理由和清理时机 |
| 是否有不可接受的技术债混入？ | 检查"不可接受"列表，确认 MVP 不含 |
| 分期是否合理？ | P0 能独立交付，P1/P2 是增量 |
| 是否与复杂度 Level 匹配？ | Level 1 不做完整 Token，Level 3 不跳过 |
