# 架构方案维护指南

> 本文件定义架构方案的存放、引用、变更和维护规则。
> 确保架构方案不是一次性文档，而是后续开发的持续参考。

---

## 一、架构方案存放位置

| 场景 | 存放路径 | 说明 |
|---|---|---|
| 新项目 | `docs/frontend-architecture.md` | 项目根目录下的 docs/ |
| 知识库项目 | 与设计书同目录 | 便于对照查阅 |
| 多项目共享 | 每个项目独立一份 | 不跨项目共享架构方案 |

**命名约定**：`frontend-architecture.md`，不使用其他名称。

## 二、后续编码如何引用

### 2.1 下游 Skill 消费路径

后续 AI coding skill 应按以下步骤消费架构方案：

```
1. 读取 YAML frontmatter（11 字段）→ 获取关键决策（框架、Register、Level、组件库、样式、状态管理）
2. 读取「架构约束」区 → 了解不可违反的工程约束
3. 按 YAML 中的 Level + Register 决定输出深度
4. 编码时遵守架构约束，不绕过已定义的方案
```

### 2.2 关键消费点

| 消费场景 | 读取内容 | 说明 |
|---|---|---|
| 创建新文件 | §5 目录结构 + §2 复杂度 Level | 文件应放在正确的目录 |
| 选择组件 | §4 组件库 + §3 技术栈 | 使用已选定的组件库 |
| 管理状态 | §9 数据与状态（L2+） | 遵循已定义的状态边界 |
| 编写样式 | §8 Design Token（L2+） | 使用 Token，不硬编码 |
| 判断 MVP 范围 | §11 MVP 范围 | 功能是否在 MVP 内 |
| 处理冲突 | 「架构约束」区 | 约束优先，冲突回到本 skill |

### 2.3 冲突处理

当下游 skill 的建议与架构方案冲突时：

1. **架构约束优先**——约束区的 6 条不可违反
2. 如果下游 skill 有更好的方案——回到本 skill 重新评估，更新架构方案
3. 不在编码阶段自行修改架构决策——先更新架构方案再编码

## 三、架构变更如何记录

### 3.1 架构决策记录

在架构方案的「架构决策记录」区维护：

| 字段 | 说明 |
|---|---|
| 日期 | 决策时间 |
| 决策 | 做了什么决定 |
| 理由 | 为什么这样做 |
| 替代方案 | 考虑过但没选的方案 |
| 触发方 | 谁做的决定（用户 / Agent / 架构评审） |

### 3.2 架构变更记录

在架构方案的「架构变更记录」区维护：

| 字段 | 说明 |
|---|---|
| 日期 | 变更时间 |
| 变更内容 | 改了什么 |
| 原因 | 为什么改 |
| 触发方 | 谁触发的变更 |

### 3.3 变更原则

| 原则 | 说明 |
|---|---|
| 小变更直接记 | 组件调整、目录微调 → 在变更记录追加一行 |
| 大变更需评审 | 框架切换、组件库更换、Level 变化 → 重新走架构设计流程 |
| 变更不影响约束 | 架构约束区的内容变更需格外慎重 |
| 记录不删除 | 历史记录保留，不删除旧条目 |

## 四、什么时候需要重新评审

| 触发条件 | 评审范围 | 说明 |
|---|---|---|
| Level 升级（L1→L2、L2→L3） | 全量重审 | Level 变化影响输出深度和架构复杂度 |
| Register 变化 | 全量重审 | Register 变化影响组件库、目录、状态方案 |
| 框架更换 | 全量重审 | 如 React→Vue，所有方案需重做 |
| 组件库更换 | §4 + §7 + §8 | 影响组件清单和 Token |
| 新增核心功能模块 | §5 + §6 + §7 + §11 | 影响目录、领域、组件、MVP |
| MVP 阶段切换（P0→P1→P2） | §11 + 技术债 | 补充新能力，清理技术债 |
| 架构约束被违反 | 约束区 + 相关章节 | 确认是否需要调整约束 |

### 重新评审流程

```
1. 确认变更范围（全量 / 局部）
2. 更新 YAML frontmatter 中受影响的字段
3. 更新受影响的章节
4. 运行 decision-checklist 自检
5. 在变更记录区追加条目
```

## 五、YAML Frontmatter 消费指南

架构方案的 YAML frontmatter 是下游 skill 的**快速入口**，11 个字段的设计目的：

```yaml
framework: Astro           # 框架 → 决定组件写法、构建方式
register: Content           # Register → 决定组件库、目录、状态策略
complexity_level: 1         # Level → 决定输出深度
decisive_dimension: 业务复杂度 # 决定性维度 → 解释为什么是这个 Level
component_library: 无头 UI   # 组件库 → 决定可用组件和样式方式
styling: CSS Variables      # 样式方案 → 决定写样式的方式
state_management: 未使用全局状态 # 状态管理 → 决定数据流架构
architecture_focus: 构建管线  # 架构侧重 → 一句话说明重点
mvp_scope: 详情页+列表页      # MVP 范围 → 一句话说明第一阶段
created: 2026-06-12         # 创建时间
updated: 2026-06-12         # 最后更新时间
```

### 下游 Skill 读取优先级

```
1. framework + register + complexity_level → 确定工作模式
2. component_library + styling + state_management → 确定技术方案
3. architecture_focus + mvp_scope → 确定范围
4. 如需深入了解 → 读正文相关章节
5. 如遇冲突 → 读「架构约束」区
```

## 六、架构方案生命周期

```
创建（本 Skill 输出）
  → 评审（用户确认 / decision-checklist 自检）
  → 编码（下游 Skill 消费 frontmatter + 约束区）
  → 变更（记录在变更记录区）
  → 阶段切换（P0→P1→P2，更新 MVP 范围）
  → 重新评审（Level/Register/框架变化时）
```

### 各阶段维护重点

| 阶段 | 维护重点 |
|---|---|
| P0（MVP） | 确保 MVP 范围准确，技术债已记录 |
| P1（可用） | 清理 MVP 技术债，补充缺失能力，更新变更记录 |
| P2（完善） | 补充 Token/主题/i18n，完善组件体系 |
| P3（优化） | 性能优化，a11y 完善，技术债清零 |

## 七、常见反模式

| 反模式 | 为什么是错的 | 正确做法 |
|---|---|---|
| 架构方案写完就扔 | 后续开发无参考，架构决策被遗忘 | 架构方案持续维护，随项目演进 |
| 编码时自行修改架构决策 | 架构方案与代码不同步 | 先更新方案再编码 |
| 不记录变更原因 | 后续无法理解决策背景 | 每次变更都记录理由 |
| YAML frontmatter 不更新 | 下游 skill 读取到过期信息 | 每次方案变更同步更新 frontmatter |
| 架构约束区随意修改 | 约束是下游 skill 的护栏 | 约束变更需重新评审 |
| 多份架构方案并存 | 不确定以哪份为准 | 一个项目只维护一份 |
