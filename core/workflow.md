# 前端架构设计 Skill — 核心工作流

> 本文件是前端架构设计能力的唯一工作流入口。
> Agent 读完后知道：先问什么、先读什么、怎么判定、怎么输出、怎么衔接。

---

## 一、能力定位

本能力用于在前端项目编码前，帮助用户完成项目级前端架构规划。

**负责**：技术选型、组件库选择、目录结构、模块边界、组件封装策略、Design Token、状态架构、主题/i18n 预留、MVP 范围规划。

**不负责**：UI 视觉设计、具体页面实现、后端架构、自动化测试专项、WebGL/Canvas 游戏/低代码平台内核。

**产出物**：一份结构化的 `frontend-architecture.md`，存入项目 `docs/` 目录，供后续 AI coding 和下游 skill 消费。

---

## 二、红旗表（Red Flags）

如果 Agent 脑中出现以下想法，**必须停下来，回到本工作流**：

| 想法 | 事实 |
|---|---|
| "这个项目很简单，不需要架构" | 简单项目也需要目录结构和组件库选择，Level 1 输出够轻 |
| "用户已经说了用什么技术栈，直接开始" | 技术栈只是架构的一环，还有目录、组件分层、Token、MVP 范围 |
| "先写个页面看看效果再调" | 先写页面再调结构 = vibe coding 的核心问题，正是本 skill 要消灭的 |
| "架构规划太费时间了，先出 MVP" | 架构规划本身就是 MVP 的一部分，输出深度与 Level 匹配，不会过重 |
| "用户没问架构，我不应该主动提" | 如果用户要求开始前端编码，架构是必要前置步骤 |
| "这个是存量项目，架构已经定了" | 存量项目更需要架构诊断，不是不需要 |

## 三、适用与不适用场景

### 适用

- 从零设计前端项目架构
- 重构存量前端项目结构
- 前端技术选型与组件库选择
- 目录结构与模块边界设计
- 为 AI coding 项目建立工程约束

### 不适用（应主动退出）

- 纯后端 / API 项目，无前端部分
- 用户只说"帮我做个网站"，给不出任何具体需求或场景
- WebGL / Canvas 游戏 / 小程序 / 低代码平台内核等专项架构
- 用户已进入具体页面编码阶段，明确不需要架构规划
- 已有组织级前端架构规范且用户要求严格照搬

如果信息不足，不应立即退出。应先问少量关键问题；如果用户仍无法提供，则输出 Level 1 草案并标注假设。

## 三·五、前置意图判断

在进入必问题之前，先判断用户意图是**全量架构设计**还是**局部问题**：

| 判断信号 | 意图类型 | 下一步 |
|---|---|---|
| 用户说"前端架构""全量设计""从零规划""前端方案" | 全量设计 | 继续到 §4 必问题 → 全流程 |
| 用户说"选组件库""目录怎么规划""状态管理""Token""MVP" 等单一话题 | 局部问题 | 直接跳到 §8 路由表 → 子工作流 |
| 用户说"重构""梳理架构""诊断" 且已有项目 | 存量项目 | 继续到 §4 必问题 → 工作流 B |
| 意图模糊 | 不确定 | 问一句："你需要完整的前端架构规划，还是只需要某个方面的建议？" |

**局部问题的捷径**：如果意图明确是局部的，只需快速判定 Level（1 个问题即可：个人原型 / 团队产品 / 长期企业级？），然后按路由表的"输出项"列输出，不全量收集 5 个必问题。

## 四、关键问题清单

Agent 不应一次性问完所有问题。先问必问题，根据回答判定复杂度和 Register，再按需追问。

### 必问（5 项）

| # | 问题 | 影响 |
|---|---|---|
| 1 | 这是新项目还是存量项目重构？ | 决定走工作流 A 还是 B |
| 2 | 项目类型：官网 / 知识库 / 后台管理 / SaaS / 桌面应用 / 数据看板 / 移动端 / 其他？ | 影响选型、Register、组件库 |
| 3 | 主要用户是谁，核心使用场景是什么？ | 影响领域与页面形态 |
| 4 | 预期规模与生命周期：个人原型（< 1 月）/ 团队产品（3-6 月）/ 长期企业级（> 6 月）？ | 影响复杂度 Level |
| 5 | 是否已有技术约束：React / Vue / Astro / Next.js / Vite / Tauri / Electron / Docker？ | 影响选型 |

复杂度判定主要取第 2、4 项。必问题回答完毕后即可判定复杂度和 Register。

### 按需追问（影响具体方案）

- 是否需要 SSR / SSG？
- 是否需要暗色模式或主题切换？
- 是否需要国际化？
- 是否有大量表格、表单、图表、编辑器或搜索？
- 是否需要权限、工作流或复杂状态管理？
- 是否需要兼容移动端或桌面端？
- MVP 第一阶段必须完成哪些功能？

### 询问原则

- 先问 5 个必问题
- 根据回答判定复杂度 Level 和 Register
- 再针对性追问，不一次全问
- 如果用户已提供设计书，先读取设计书，再补充缺失信息

## 五、复杂度模型

### 判定规则

对「团队规模、生命周期、业务复杂度」三个维度**各自定级，最终取最高级**。

| 维度 | → Level 1 | → Level 2 | → Level 3 |
|---|---|---|---|
| 团队规模 | 1 人 | 3-5 人 | 10+ 人 |
| 生命周期 | < 1 个月 | 3-6 个月 | > 6 个月 |
| 业务复杂度 | 原型验证 / 内部工具 | 明确业务模型和用户场景 | 多业务域 / 复杂权限 / 多端适配 |

**仲裁示例**：1 人（→ L1）+ 长期 > 6 月（→ L3）→ 判 **Level 3**。

Agent 必须在输出的「复杂度判定」一节**显式写明取级依据和决定性维度**。

### 决定性维度与架构侧重

输出复杂度判定时，必须同时输出决定性维度和对应的架构侧重方向：

| 决定性维度 | 架构侧重 |
|---|---|
| 团队规模驱动的高 Level | 加强模块边界、命名规范、组件文档化 |
| 生命周期驱动的高 Level | 加强可演进性、技术债管理、架构决策记录 |
| 业务复杂度驱动的高 Level | 加强领域拆分、状态架构、数据流设计 |

### Level 对输出范围的影响

| # | 输出项 | L1 | L2 | L3 |
|---|---|:--:|:--:|:--:|
| 1 | 项目理解摘要 | ✅ | ✅ | ✅ |
| 2 | 复杂度判定（含决定性维度 + 架构侧重） | ✅ | ✅ | ✅ |
| 3 | 推荐技术栈 | ✅ | ✅ | ✅ |
| 4 | UI 组件库选择与理由 | 简 | ✅ | ✅ |
| 5 | 目录结构方案 | ✅ | ✅ | ✅ |
| 6 | 模块边界与领域拆分 | — | 简 | ✅ |
| 7 | 核心组件清单 | ✅ | ✅ | ✅ |
| 8 | Design Token 初稿 | — | ✅ | ✅ |
| 9 | 数据与状态架构 | — | ✅ | ✅ |
| 10 | 主题与国际化预留 | — | — | ✅ |
| 11 | MVP 第一阶段范围 | ✅ | ✅ | ✅ |
| 12 | 后续编码建议 | ✅ | ✅ | ✅ |
| — | 架构决策记录（追加区） | — | ✅ | ✅ |

「简」= 给方向不展开；「—」= 该 Level 不输出。

### Level 对 Reference 加载的影响

- **Level 1**：只读必需 reference，跳过 `domain-boundaries`、`state-and-data`、`theme-and-i18n`
- **Level 2**：读取相关 reference，部分内容可简化
- **Level 3**：读取全部相关 reference，展开关键取舍

### 过度设计防护

- Level 1 项目不应输出领域拆分、完整 Token、主题国际化
- Level 1 项目的目录结构应保持扁平，不强行 feature-based
- 所有 Level 都应先问清楚再输出，不默认套企业架构模板

## 六、Register 架构分轨

在复杂度判定之后、reference 路由之前，判定项目的 **Register**（架构策略分轨）。

### Register 判定

| Register | 判定信号 | 典型项目 |
|---|---|---|
| **Content** | 内容展示为主、SEO 重要、页面偏静态、用户以浏览为主 | 知识库、博客、文档站、官网、作品集 |
| **Tool** | 交互操作为主、数据管理密集、用户以操作为主 | SaaS、后台管理、数据看板、工作流平台 |
| **Desktop** | 运行在桌面环境、需要本地能力 | Tauri / Electron 应用 |
| **Hybrid** | 既有内容展示又有工具操作，需要区分策略 | 带管理后台的内容平台、带文档的 SaaS |

如果模糊，优先问一句："这个项目更偏内容展示（用户主要浏览）还是工具操作（用户主要做事）？"

### Register 对架构策略的影响

| 维度 | Content | Tool | Desktop |
|---|---|---|---|
| 渲染策略 | 优先 SSR/SSG | 默认 CSR，按需 SSR | 本地渲染 |
| 组件库方向 | 轻量（Astro 组件 / 无头 UI） | 生态完整（Ant Design / MUI / shadcn） | 现代轻量（shadcn/ui） |
| 目录结构 | 按内容类型组织 | 按功能模块 / feature 组织 | 按功能模块 + 本地桥接层 |
| 是否需要 DDD | 通常不需要 | Level 2+ 可能需要 | 视复杂度而定 |
| 状态管理 | 轻量或不需要 | 标准方案（TanStack Query + Zustand） | 可能需要本地状态持久化 |
| 典型框架 | Astro / Next.js SSG | React + Vite / Next.js | Tauri + React |

### Register 反模式

| Register | 不应该 |
|---|---|
| Content | 推荐 Ant Design / Element Plus 作为主组件库；输出 DDD 领域拆分；设计成复杂后台应用 |
| Tool | 使用 Astro 做管理后台；完全不用组件库而全部手写；忽视表格/表单/权限场景 |
| Desktop | 忽视本地集成策略（窗口管理、文件系统、离线）；推荐纯 Web 部署方案 |

## 七、Architecture Read（架构阅读）

**在正式输出架构方案前，必须先声明理解。这是一个非可选步骤。**

格式：

> *"理解本项目为 **[新/存量]** **[Register]** 项目，核心用户是 **[谁]**，主要场景是 **[什么]**，复杂度判定为 **Level [X]**（决定性维度：**[维度]**，架构侧重：**[方向]**），主要架构挑战是 **[什么]**。"*

**如果用户未纠正**：继续输出架构方案。

**如果用户纠正**：重新判定后再输出。不得带着已知错误理解继续。

这一步的成本是一句话，但能避免最大的浪费：基于错误理解产出的完整架构方案。

## 七点五、AI 友好命名四原则

所有目录、组件、hook、API、store、token、主题 key 和 i18n key 命名都遵循同一组原则：

| 原则 | 说明 | 通过标准 | 问题信号 |
|---|---|---|---|
| 可搜索 | 全局搜索名称能稳定命中目标 | 文件名、变量名、key 具备业务语义 | `Form.tsx`、`api.ts`、`--c1` |
| 语义明确 | 名字即文档，看到名称能判断职责 | 包含实体、动作或用途 | `Badge1`、`useData`、`mode1` |
| 遵循约定 | 同类对象使用统一命名模式 | 组件 PascalCase，hook camelCase + `use`，token 语义化 | 同一类文件混用多种模式 |
| 模式稳定 | 项目内长期保持一致 | 新增命名能套入现有模式 | 每个模块临时发明一套命名 |

各 reference 可在此基础上补充领域专用命名模式，不重复定义四原则表。

## 九、工作流 A：新项目

适用于从零创建前端项目。

```
用户提需求 → 5 必问题 → 复杂度判定 → Register 判定
  → Architecture Read → [用户确认]
  → 路由表加载 references → 输出架构方案
  → decision-checklist 自检 → 后续步骤建议
```

### 步骤 1：项目理解

- 阅读用户提供的项目说明、设计书或现有代码结构
- 识别项目类型、用户角色、核心使用场景和主要页面形态
- 明确前端偏展示型、工具型、管理型、编辑型或混合型

### 步骤 2：复杂度判定

- 按必问题第 2、4 项确认 Level
- 三维度各自定级，取最高级
- 记录决定性维度和架构侧重

### 步骤 3：Register 判定

- 按必问题第 2、3 项确认 Register
- 如果模糊，问一句区分

### 步骤 4：Architecture Read

- 输出一句话声明理解
- 等待用户确认或纠正

### 步骤 5：Reference 路由

- 根据意图确定子工作流（完整 or 局部）
- 根据子工作流 + Level + Register 查表加载 references
- Level 1 跳过 `domain-boundaries`、`state-and-data`、`theme-and-i18n`

### 步骤 6：技术栈与 UI 组件库

- 按 Register + Level 推荐框架、构建工具、样式方案、组件库和图标库；细则见 §8 路由表与 `ui-library-selection.md`

### 步骤 7：目录结构与模块边界

- 按 Level 选择 L1 扁平 / L2 feature-based / L3 领域化结构，并按 Register 调整；读取 `directory-structure.md`，Level 2+ 按需读取 `domain-boundaries.md`

### 步骤 8：组件体系

- 识别应用壳、业务组件、通用组件和封装边界；读取 `component-architecture.md`

### 步骤 9：数据与状态架构

- Level 2+ 明确 server-state / client-state / local-state 边界、数据获取层和全局状态方案；读取 `state-and-data.md`

### 步骤 10：Design Token、主题与国际化

- Level 2+ 输出 Token 体系；Level 3 进一步输出主题与 i18n 预留；读取 `design-tokens.md`，Level 3 读取 `theme-and-i18n.md`

### 步骤 11：MVP 规划

- 明确第一阶段页面范围、必须封装组件、延后能力和技术债；读取 `mvp-planning.md`

### 步骤 12：输出与自检

- 使用 `core/output-template.md` 的结构输出方案
- 产出路径：目标项目已有目录 → `{项目根}/docs/frontend-architecture.md`；无真实项目目录（试跑/诊断） → skill 包 `outputs/{项目名}-frontend-architecture.md`；用户指定路径优先
- 使用 `core/decision-checklist.md` 自检
- 如果自检发现问题，修正后重新输出

## 十、工作流 B：存量项目

适用于已有前端项目，要求重新梳理架构或做渐进式重构。

```
用户提需求 → 5 必问题 → 复杂度判定 → Register 判定
  → 结构化上下文采集 → 现状诊断 → Architecture Read
  → [用户确认] → 增量改造方案 → decision-checklist 自检
```

### 步骤 1：项目理解

同工作流 A 的 5 必问题。

### 步骤 2：复杂度判定

同工作流 A。额外记录：当前项目处于哪个 Level。

### 步骤 3：Register 判定

同工作流 A。

### 步骤 4：结构化上下文采集

**前 3 分钟必做，按优先级读取（存在就读，不存在就跳过）**：

| 优先级 | 读取目标 | 目的 |
|---|---|---|
| 1 | `package.json` | 现有技术栈和依赖 |
| 2 | 目录树（顶层 2-3 层） | 现有组织方式 |
| 3 | `tsconfig.json` / `vite.config.*` / `next.config.*` | 构建配置 |
| 4 | `src/App.tsx` / `src/main.ts` / `pages/` | 入口和路由结构 |
| 5 | 任意 2-3 个组件文件 | 组件风格和封装程度 |
| 6 | `tailwind.config.*` / 全局 CSS 文件 | 样式方案 |

### 步骤 5：现状诊断

- 对比现有结构与推荐结构的差异
- 识别高风险问题（业务逻辑混入组件、样式硬编码、全局状态滥用）
- 识别低风险问题（命名不规范、目录层级可优化）
- 识别已有优势（做得好的地方应保留）

### 步骤 6：Architecture Read

声明对现有项目的理解和改造方向。

### 步骤 7：增量改造方案

输出内容：

- **当前结构诊断**：问题分类（高风险 / 低风险 / 优势保留）
- **保留项**：现有做得好、不应动的部分
- **调整项**：低成本高收益的改进
- **迁移步骤**：分步改造，每步可独立验证
- **风险与回滚**：每步的风险和回滚方案
- **技术债记录**：暂不处理但需记录的项

**核心原则**：除非用户明确要求，否则**不建议全量重构**。

### 步骤 8：自检

同工作流 A。

## 十一、信息不足时的处理

| 情况 | 处理方式 |
|---|---|
| 用户给不出项目类型 | 问一句："大概是一个什么类型的产品？"如果仍无法回答，按 Level 1 输出并标注假设 |
| 用户给不出技术约束 | 推荐一个默认方案，标注"基于无约束假设，如有约束请告知" |
| 用户拒绝回答必问题 | 不追问，按最保守判定输出 Level 1 草案，标注所有假设 |
| 用户提供的描述与 Register 判定矛盾 | 以用户描述为准，不强行归入某个 Register |
| 用户在 Architecture Read 后纠正 | 回到 §4 必问题 + §5 复杂度 + §6 Register 全量重判，再走一遍输出流程 |

## 十二、后续步骤与 Skill 协作

架构方案输出完成后，根据项目需要，建议执行以下后续步骤。
**每一步都有推荐的 skill 和 fallback 方案。委托不是强制的。**

### 委托点 1：视觉设计

**触发条件**：项目需要具体的页面视觉设计。

| 场景 | 推荐 | Fallback |
|---|---|---|
| 工具型 / 后台 UI | Impeccable `craft` 或 `shape` | 架构方案中补充基础视觉规范 |
| 落地页 / 作品集 / 营销页 | Taste Skill | 架构方案中补充三表盘参数建议 |
| 已有品牌规范 | Impeccable `init` | 架构方案中记录品牌约束 |

协作要求：视觉设计 skill 应先读 `frontend-architecture.md`；视觉设计在架构约束内执行；冲突时回到本 skill。

### 委托点 2：项目脚手架

**触发条件**：架构方案确认后，准备创建项目骨架。

| 场景 | 推荐 | Fallback |
|---|---|---|
| React 项目 | `create-next-app` / `create vite` + 调整 | 手动创建目录 |
| Vue 项目 | `create vue` + 调整 | 手动创建 |
| Astro 项目 | `create astro` + 调整 | 手动创建 |

协作要求：脚手架输出应按架构方案调整；组件库/样式/状态管理应在脚手架阶段配置。

### 委托点 3：编码流程管理

**触发条件**：进入具体功能开发阶段。

| 场景 | 推荐 | Fallback |
|---|---|---|
| 有 Superpowers | brainstorming → plan → TDD 流程 | 架构方案中的"后续编码建议" |
| 没有 | 遵循架构约束逐功能实现 | 同左 |

协作要求：持续引用 `frontend-architecture.md`；每个功能开发前检查 MVP 范围；架构变更记录在"架构决策记录"区。

### 委托点 4：存量项目诊断深化

**触发条件**：存量项目需要深度质量审计。

| 场景 | 推荐 | Fallback |
|---|---|---|
| 有 Impeccable | `audit` 命令 | 本 skill 的 decision-checklist 自检 |
| 有 Superpowers | `systematic-debugging` | 手动排查 |

### 通用协作原则

1. **委托不是强制的**——用户可以跳过任何委托点
2. **每个委托点都有 fallback**——不安装推荐 skill 也不影响核心流程
3. **架构方案是共享契约**——所有下游 skill 都应先读 `frontend-architecture.md`
4. **架构约束优先**——下游 skill 建议与架构冲突时，回到本 skill 重新评估
5. **架构变更要记录**——任何修改都记录在"架构决策记录"区
