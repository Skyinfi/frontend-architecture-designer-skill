# 决策自检清单

> 本文件供架构方案输出前的自检使用。Agent 应在输出方案后逐项检查，确保无缺项、无过度设计、无反模式。
> 不依赖某个 Agent，所有平台通用。

---

## 一、技术栈检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | 框架选择是否与 Register 匹配？ | Content → Astro；Tool → React/Vue；Desktop → React+Tauri | Content 推了 SPA 框架 |
| 2 | 技术栈是否尊重已有约束？ | 设计书/用户指定的约束全部采纳 | 忽略用户指定的框架/库 |
| 3 | 样式方案是否与组件库协调？ | 组件库自带样式则不引入冲突方案；无头 UI 则配 Tailwind/Token | 同时引入 Tailwind + Ant Design 内置样式 |
| 4 | 构建工具是否合理？ | Astro/Vite（轻量）；不为了用工具而用工具 | Content 项目用 Webpack 自定义配置 |
| 5 | 是否引入了不需要的依赖？ | 每个依赖都有明确理由 | "先装上以后可能用" |

---

## 二、目录结构检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | 目录结构是否与 Level 匹配？ | L1 扁平 / L2 feature-based / L3 领域化 | L1 用了领域化结构 |
| 2 | 每个目录职责是否清楚？ | 放什么/不放什么有明确说明 | 目录职责模糊 |
| 3 | 通用与业务是否分开？ | shared/components/（通用）+ features/（业务） | 所有组件混在一个目录 |
| 4 | 是否有 docs/ 目录？ | 架构方案有固定存放位置 | 架构方案无处存放 |
| 5 | 构建管线目录是否合理？ | scripts/ 独立；Content Collection 在 src/content/ | 构建脚本散落在各处 |

---

## 三、组件体系检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | 三层分离是否清晰？ | UI / 业务 / 页面 组件职责不重叠 | UI 组件包含 API 调用 |
| 2 | 应用壳组件是否确定？ | AppShell/Layout 已列入清单 | 无全局布局方案 |
| 3 | 组件库选择是否与 Register 匹配？ | Content → 无头/极轻量；Tool → 企业级；Desktop → shadcn/ui | Content 推了 Ant Design |
| 4 | 组件命名是否 AI 友好？ | 可搜索、语义明确、遵循约定、模式稳定 | `Comp1`、`Wrapper`、`Form2` |
| 5 | 是否避免了过早抽象？ | 无只出现 1 次就被封装的组件 | 10+ 个只用到 1 次的"通用"组件 |

---

## 四、数据与状态架构检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | server-state / client-state 是否分离？ | API 数据用数据获取层；UI 状态用状态管理 | 全部放 useState 或全局 store |
| 2 | 数据获取层是否确定？ | 有明确方案（TanStack Query / SWR / 构建时） | 散落的 fetch/axios 调用 |
| 3 | 是否过度使用全局状态？ | 只有真正全局的 UI 状态放全局 | 表单值、弹窗开关放全局 |
| 4 | Level 1 是否被强行引入状态管理？ | L1 标注"未使用全局状态" | L1 引入了 Zustand/Redux |
| 5 | 请求层是否统一封装？ | 有 shared/request/ 或等价结构 | 每个组件各自 try-catch |

## 五、Token 与主题检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | Level 1 是否被输出完整 Token？ | L1 标注"暂不展开" | L1 输出了 7 类完整 Token |
| 2 | Token 是否用语义化命名？ | `--color-primary` 而非 `--blue-500` | 按色值命名 |
| 3 | 主题切换方案是否合理？ | CSS Variables + data-theme | JS 切换 className |
| 4 | 是否与组件库 Token 冲突？ | 自定义 Token 与组件库 Token 协调 | 两套体系互不兼容 |

## 六、MVP 范围检查

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | MVP 是否构成完整闭环？ | 核心用户流程从头到尾可走通 | 缺了关键步骤 |
| 2 | 是否有明确延后的能力？ | P0/P1/P2 分期清晰 | "全部 P0" |
| 3 | 技术债是否记录？ | 每个已知妥协有理由和清理时机 | 技术债被忽略 |
| 4 | 不可接受的技术债是否为零？ | MVP 不含安全/架构底线问题 | 有"不可接受"项混入 MVP |
| 5 | MVP 范围是否与 Level 匹配？ | L1 不做 Token/主题/i18n/领域拆分 | L1 做了完整设计系统 |

## 七、存量项目迁移检查

> 仅存量项目需要。新项目跳过。

| # | 检查项 | 通过标准 | 问题信号 |
|---|---|---|---|
| 1 | 是否先采集了现有结构？ | 读了 package.json/目录树/构建配置等 | 未读现有代码直接出方案 |
| 2 | 是否尊重现有技术栈？ | 保留可用方案，增量改造 | 忽略现有栈推荐全量替换 |
| 3 | 是否避免全量重构建议？ | 增量改造 + 迁移步骤 | "建议重写整个项目" |
| 4 | 保留项是否明确？ | 列出了保留什么、为什么 | 所有现有方案都被否定 |
| 5 | 迁移步骤是否可渐进执行？ | 每步可独立验证，有回滚方案 | 只有一个大步骤"迁移全部" |

## 八、质量基线提示项（F9）

> 以下项在 MVP 阶段"提一句、留接口"即可，不要求深入做。但架构方案中应至少有一行提到它们，不能只字不提。

| # | 提示项 | 架构方案中应包含 | MVP 阶段建议 |
|---|---|---|---|
| 1 | 可访问性（a11y）基线 | 是否需要？如果需要，基线是什么？ | 提一句：组件用语义化 HTML + 基础 aria 属性 |
| 2 | 性能 / bundle 预算 | 是否设预算？首屏指标？ | 提一句：Astro 默认零 JS / React bundle < 200KB |
| 3 | 测试策略 | 是否需要？E2E 还是单元测试？ | 提一句：核心流程 E2E，关键函数单元测试 |
| 4 | 错误监控 | 是否需要？Sentry / 自建？ | 提一句：至少有全局错误兜底 |
| 5 | CI/CD | 是否需要？GitHub Actions？ | 提一句：手动构建可部署，后续加 CI |

## 九、Architecture Anti-Slop 负面检查区

> 机械验证 AI 最易犯的错误。**任何一项命中 = 方案有问题，必须修改。**

| # | 检查项 | 命中条件 | 修正方向 |
|---|---|---|---|
| 1 | 是否对所有项目都推 React + shadcn/ui？ | Content 项目推了 React SPA | Content → Astro / 静态方案 |
| 2 | 是否对简单 CRUD 输出完整 DDD？ | L1 项目有领域拆分章节 | L1 标注"暂不展开" |
| 3 | 是否对静态内容站推 Ant Design / Element Plus？ | Content Register 推了重型组件库 | Content → 无头/极轻量 |
| 4 | 是否对存量项目建议全量重构？ | 存量项目方案以"重写"开头 | 增量改造 + 迁移步骤 |
| 5 | 是否忽略已有技术约束？ | 用户指定了框架但方案用了另一个 | 以用户约束为准 |
| 6 | 是否对局部问题输出完整 12 项？ | 用户只问"组件库选什么"但输出全部 12 项 | 走子工作流，只输出相关项 |
| 7 | 是否给 Level 1 输出完整 Token？ | L1 方案包含 7 类 Token | L1 标注"暂不展开" |
| 8 | 是否推荐互相冲突的组件库？ | 同时推荐 Ant Design + Element Plus | 选一套 |
| 9 | 是否在 MVP 里做完整设计系统？ | MVP 包含完整 Token + 主题 + i18n | MVP 只做核心闭环 |
| 10 | 是否对桌面应用忽略本地集成？ | Desktop Register 方案无文件系统/托盘相关 | 补充本地集成组件 |

---

## 十、自检使用方式

1. 架构方案输出后，按以上九节逐项检查
2. 每项标记 ✅（通过）或 ❌（需修改）
3. ❌ 项必须在方案中修正后重新检查
4. Anti-Slop 区（第九节）任何一项命中 = 方案不可通过
5. 质量基线提示项（第八节）只要求"提到"，不要求"做到"
